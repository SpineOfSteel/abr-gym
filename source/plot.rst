Plots and Evaluation
====================

This report consolidates the earlier plot pages into one easy-to-follow ``plot.rst`` file. It starts with the reinforcement-learning training curves, introduces each algorithm briefly, and then moves into the transport-wise evaluation plots for the Norway mobile bandwidth dataset.

.. contents:: Table of contents
   :local:
   :depth: 2

Overview
--------

The figures documented here come from two complementary parts of the project:

1. **RL training curves** for ABR agents trained in the custom ``abr-gym`` environment using Stable-Baselines3 log directories.
2. **Transport-wise evaluation plots** generated by ``plot_grouped.py`` for the Norway MMSys 2013 commute-path bandwidth dataset.

Together, these plots tell a complete ABR story: first, which learning algorithm converges fastest and strongest, and second, how the final ABR schemes behave on real mobility traces under bus, car, ferry, metro, train, and tram conditions.

RL Training Curves
------------------

This section compares the six reinforcement-learning agents trained from monitor logs stored in separate directories:

.. code-block:: python

   log_dir_dqn = 'logs/dqn/'
   log_dir_sac = 'logs/sac/'
   log_dir_a2c = 'logs/a2c/'
   log_dir_ppo = 'logs/ppo/'
   log_dir_ddpg = 'logs/ddpg/'
   log_dir_td3 = 'logs/td3/'

   log_dirs = [
       log_dir_ppo,
       log_dir_dqn,
       log_dir_sac,
       log_dir_a2c,
       log_dir_ddpg,
       log_dir_td3,
   ]

The plots were generated with the Stable-Baselines plotting helper:

.. code-block:: python

   plt.figure(figsize=(20, 10))
   abr.plot_results(log_dirs, 50_000, X_WALLTIME, "All RL Models")
   plt.legend(['PPO', 'DQN', 'SAC', 'A2C', 'DDPG', 'TD3'], ncol=6, loc='lower right')
   plt.tight_layout()
   plt.show()

One important detail is that one supplied image legend says **A3C**, but the actual code uses ``log_dir_a2c = 'logs/a2c/'``. So the red run should be interpreted as **A2C**, unless that directory was intentionally reused for an A3C experiment.

Short explanation of each RL algorithm
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

**PPO**
   Proximal Policy Optimization is a stable on-policy actor-critic method. It is a strong ABR baseline because it usually learns steadily and avoids the brittleness of older policy-gradient approaches.

**DQN**
   Deep Q-Network is an off-policy value-based algorithm for discrete action spaces. In ABR, it fits naturally when each action directly selects a bitrate level from a finite ladder.

**SAC**
   Soft Actor-Critic is an off-policy actor-critic algorithm with entropy regularization. It tends to explore well and learn robust policies, which often helps in ABR control problems with noisy network behavior.

**A2C**
   Advantage Actor-Critic is a synchronous actor-critic baseline. It is simpler than PPO, but often weaker in sample efficiency and final performance.

**DDPG**
   Deep Deterministic Policy Gradient is an off-policy continuous-control algorithm. In ABR, it can learn a continuous control signal that is later mapped back into a bitrate decision.

**TD3**
   Twin Delayed DDPG improves DDPG by reducing overestimation and stabilizing critic learning. It is frequently one of the strongest continuous-control baselines.

Why these curves matter for ABR
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

An ABR controller is not judged only by average bitrate. It must learn to balance three competing goals:

- maximize delivered video quality,
- minimize rebuffering,
- avoid abrupt bitrate oscillation.

The training curves therefore show how quickly each RL algorithm discovers a useful quality–stall–smoothness tradeoff.

Reward comparison by wall-clock time
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

   :alt: RL model reward comparison versus walltime
   :width: 95%
   :align: center

   Episode reward versus wall-clock training time for PPO, DQN, SAC, A2C, DDPG, and TD3.

The wall-time plot answers a practical question: which model reaches a good policy fastest in real elapsed time? DDPG and TD3 climb almost immediately to the top reward band near 11k, while SAC also improves very quickly but settles slightly lower. PPO starts much lower and needs more elapsed time to approach the top cluster. DQN improves in stages, showing slower early learning than the best continuous-control methods. The red curve, labeled A3C in the supplied figure but sourced from ``logs/a2c/``, plateaus early around 7k and clearly underperforms the others.

For ABR, this suggests that replay-based actor-critic methods with continuous optimization are especially effective when the goal is to discover a strong bitrate-control policy quickly.

Reward comparison by episodes
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

   :alt: RL model reward comparison versus episodes
   :width: 95%
   :align: center

   Episode reward versus training episodes for PPO, DQN, SAC, A2C, DDPG, and TD3.

The episode plot removes raw clock-speed differences and instead compares learning as more rollouts are collected. DDPG and TD3 still dominate early, which points to strong sample efficiency on this environment. SAC reaches high reward quickly but with a slightly lower ceiling. DQN makes a noticeable mid-training jump, suggesting delayed but meaningful value improvement. PPO improves steadily and predictably. The A2C/A3C-labeled curve saturates much earlier than the others.

For ABR, this means that the best continuous-control agents do not merely train faster computationally; they also extract more useful learning signal from each batch of interaction.

Reward comparison by timesteps
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

.. figure:: timesteps.png
   :alt: RL model reward comparison versus environment timesteps
   :width: 95%
   :align: center

   Episode reward versus environment timesteps for PPO, DQN, SAC, A2C, DDPG, and TD3.

The timestep view is often the fairest comparison because it aligns models by environment interaction budget. The ranking remains consistent: DDPG and TD3 are strongest overall, SAC is highly competitive, PPO steadily climbs into the upper-middle band, DQN improves later, and the A2C/A3C-labeled run remains the weakest.

For ABR, the takeaway is clear: when interactions are limited, TD3, DDPG, and SAC are the most promising starting points for learning a policy that balances quality, rebuffering, and smoothness.

RL takeaway
~~~~~~~~~~~

Taken together, the three RL panels tell a consistent story. PPO is stable and respectable, DQN is workable but slower to mature, SAC is strong and robust, and DDPG/TD3 are the top performers in this setup. The weakest run is the red ``logs/a2c/`` curve, so it should be treated as a lower baseline rather than a target. For a modern ABR study, these results strongly motivate using TD3, DDPG, or SAC as the primary RL baselines, with PPO retained as a reliable on-policy reference.

Transport-wise Evaluation on the Norway Dataset
-----------------------------------------------

The remaining figures document the transport-wise comparison plots generated by ``plot_grouped.py`` for the Norway mobile bandwidth dataset. The script groups logs by transport mode, parses each scheme log, computes per-trace summary metrics, and emits one set of comparison plots for the aggregate benchmark plus one set for each transport in ``TRANSPORTS = ['tram', 'car', 'bus', 'ferry', 'metro', 'train']``.

The evaluated schemes are ``bb``, ``bola``, ``mpc``, ``rl``, ``ppo``, and ``netllm``, rendered in legends as **BBA**, **BOLA**, **RobustMPC**, **Pensieve**, **Pen-PPO**, and **NetLLM**.

Dataset background
~~~~~~~~~~~~~~~~~~

The traces come from the MMSys 2013 Norway mobile streaming dataset. The dataset organizes measurements by route and transport type and provides route maps as well as measured bandwidth profiles. Measurements were collected during TCP streaming sessions over Telenor's 3G/HSDPA network in Norway using fixed 2-second video segments. Some routes include very weak-signal regions, including tunnel-heavy metro stretches, which helps explain abrupt bitrate drops, high stall time, and major QoE differences across controllers.

A useful reading order for each transport section is:

#. the route map,
#. the measured bandwidth profile,
#. the ABR outcome plots for bitrate, smoothness, stall, QoE, and bitrate-vs-stall tradeoff.

How to read the transport plots
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

**Average bitrate per trace**
   The bitrate plot shows mean delivered bitrate. Higher is better only if it does not come with heavy stalls or violent switching.

**Bitrate smoothness per trace**
   The smoothness plot measures mean absolute bitrate change. Lower means a more stable viewing experience.

**Time spent on stall per trace**
   The stall plot reports rebuffer burden. Lower is better, and spikes indicate severe playback failure.

**QoE CDF**
   The QoE cumulative distribution is the best single ranking plot. Curves farther to the right indicate better reward over more traces.

**Bitrate vs stall tradeoff**
   This plot places mean bitrate on the x-axis and mean stall ratio on the y-axis. The ideal operating region is the bottom-right: high bitrate with low stall.

Implementation notes from ``plot_grouped.py``
---------------------------------------------

Several implementation details directly affect how these figures should be interpreted:

- ``collect_scheme_metrics()`` searches the trace root recursively for ``*.txt`` files and filters transport-specific runs by matching ``_{transport}_`` in the filename stem.
- The per-scheme accumulator is initialized as ``{'reward': [], 'bitrate': [], 'stall': [], 'smoothness': []}``, so every plot is ultimately driven by per-trace summary values rather than raw per-chunk sequences.
- ``parse_log_file()`` reads timestamp, bitrate, rebuffer duration, and reward from each line. Bitrate is converted from kbps to Mbps immediately with ``float(sp[1]) / 1000.0``.
- Malformed rows are skipped with ``try/except``, which makes the parser tolerant to noisy logs but also means partial files can silently contribute fewer samples.
- ``summarize_metrics()`` drops logs with empty bitrate lists or reward lists of length ``<= 1``.
- ``mean_reward`` is computed as ``np.mean(reward[1:])``, so the first reward entry is intentionally omitted.
- ``stall_pct`` is normalized as ``sum(rebuffer[1:]) / (VIDEO_LEN * 4.0 + sum(rebuffer[1:])) * 100``. The stall plot therefore shows a normalized stall ratio, not raw stall seconds.
- ``smoothness`` is computed as ``mean(abs(diff(bitrate)))``. It measures switching magnitude, not switch count.
- ``group_plots()`` writes figures using the pattern ``baselines-{suffix}-*.png``, where ``suffix`` is either a transport name or ``all``.
- The aggregate ``all`` plots use ``skip = 3`` while transport-specific plots use ``skip = 1``, so the global views are downsampled slightly for readability.
- ``plot_qoe_cdf()`` builds QoE curves from a histogram and cumulative sum, so the CDF is an empirical summary over per-trace rewards.

Final practical takeaway
------------------------

The report supports two main conclusions.

First, in the RL training stage, TD3, DDPG, and SAC are the strongest choices in this environment. They learn faster, reach higher rewards earlier, and appear better suited to optimizing ABR's multi-objective reward.

Second, in the mobility-trace evaluation stage, QoE and bitrate-vs-stall tradeoff are more trustworthy than bitrate alone. Several schemes can look good on mean bitrate while still producing much worse playback due to stall. Across mixed real-world traces, methods such as Pen-PPO, Pensieve, RobustMPC, and in some route groups NetLLM, look better precisely because they balance quality with robustness rather than chasing bitrate aggressively.
